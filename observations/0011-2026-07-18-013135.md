## Divergent findings

- **r0** — `drift todo`/`drift list` report spec-side locations as `main.drift.xml:0` (markers report real line ranges). Cheap to fix since the XML is already parsed. Also suggested a `drift reset --explain <marker> <spec>` preview and promoting `drift diff --all` earlier in `drift skill`.
- **r2** — Suggested a guarded bulk reset (`drift reset --all --i-have-reviewed`) to remove the temptation to parallelize per-edge resets, plus a post-reset confirmation hash. Also flagged a missing trap-case worked example in `drift skill`.
- **r3** — Bare `drift diff` (no args) prints only the usage banner; suggested either defaulting to `--all` or printing a guided hint that points at `drift todo` for candidate edges.
- **r6** — `drift diff validate main.format` (mismatched, unrelated spec/marker IDs) was accepted without warning. Should reject or warn on cross-edge ID pairs.
- **r8** — Proposed an optional `--reason "<text>"` flag on `drift reset` to record *why* each edge was resolved (audit trail + nudge toward articulating the judgment call).
- **r9** — `drift todo`'s headline counts ("2 markers… 1 spec item…") are misreadable as N+M independent todos; suggested a single edge-count headline. Also proposed wiring `drift diff` into todo output via `drift todo --diff` (or an interactive yes-prompt) to reduce mechanical steps while preserving the review-before-resolve discipline.

## Prioritized recommendations (consolidated)

1. **[High]** Make `drift reset` concurrency-safe and atomic on `.drift/state.xml`: take an advisory file lock (flock) around the entire read-modify-write, write to a temp file and `os.Rename`, and fail fast with an explicit "another drift process holds the lock" error rather than silently corrupting state. — r1, r2, r4
2. **[High]** Make `drift reset`'s success message truthful: either (a) re-verify state with an internal `todo` check before printing "Baseline updated.", or (b) print the old → new baseline hash so silent no-ops and stale-state-after-reset are impossible to mistake for real updates. — r1, r2, r4
3. **[High]** Eliminate PATH friction at onboarding: either install `drift` to `/usr/local/bin` (or a `bin/` shim via `drift init`), or add a one-line note to the task prompt and `drift skill` header stating "if `drift` is not on PATH, use `./drift`." Affects 6/10 runs. — r0, r1, r3, r6, r7, r8
4. **[Medium]** Promote per-edge classification to the leading position in `drift todo`/`drift diff` output: leading tags like `[marker-changed]` / `[spec-changed]` / `[both-changed]`, plus a verdict line on diff (e.g., "behavior-preserving change; safe to reset" vs. "behavioral change detected"). — r5, r7, r8, r9

## Next steps

The tool is functionally correct and every subject succeeded — there are no subject-side failures to chase. The actionable work is tool-level and clusters into three tiers:

**Tier 1 (correctness bug — fix first):** Recommendations 1 and 2. `drift reset` is the only state-mutating command in the common workflow and it has two distinct reliability defects: a lost-update race under concurrent invocation (r1, r2) and a misleading success message that doesn't match persisted state (r1, r4). Both are silent — subjects only noticed because they re-ran `drift todo`. A less careful agent would commit a corrupted baseline. Recommended concrete fix: flock on `state.xml` for the duration of read-modify-write, atomic-rename the new state, and re-run an internal `todo` check before printing "Baseline updated." Print the old → new hash pair in the success message.

**Tier 2 (onboarding — fix alongside):** Recommendation 3. Six of ten runs hit `command not found` on first invocation. This is trivial to remove — either install the binary on PATH in the eval harness or add a one-line `./drift` hint to the task prompt and `drift skill` header. Worth doing immediately because it's a pure friction cost with no design tradeoff.

**Tier 3 (UX clarity — schedule next):** Recommendations 4, 5, 6. The recurring UX theme is that `drift` deliberately offloads judgment to the agent (a design choice subjects praised) but doesn't surface enough structured signal to make that judgment *easy*. Concretely: (a) classify edges at a glance via leading tags in `todo` and a verdict line in `diff`; (b) emit a non-blocking contradiction hint when marker changes look like they invalidate spec text; (c) restructure `drift skill` with a quickstart + explicit source-of-truth rule + trap worked example. These are independent improvements and can be sequenced in any order.

Recommendations 7–14 are low-priority polish items from single runs; triage opportunistically. Two of them (reject mismatched spec/marker IDs in `diff` — r6; surface real spec line numbers — r0) are cheap correctness-of-output fixes that pair naturally with Tier 3.

A note on batch design: this batch used a single task across all 10 runs, which gives high confidence about *workflow* reliability but limited signal on *task-discrimination* difficulty (every subject already passed). Future batches should mix task types (level-1, level-3, multi-edge with several traps, ambiguous source-of-truth cases) to stress the discrimination capabilities the tool is meant to enforce.
