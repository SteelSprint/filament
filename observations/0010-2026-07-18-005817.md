## Divergent findings

- **r0, r3, r4, r7** — Subject ran `drift diff` *twice* before reset (once to inspect the pre-change in-sync state or after editing the code, then again after updating the spec). The other six runs ran `drift diff` exactly once before reset. Both patterns are acceptable; the doubled-diff behavior indicates subjects treat `diff` as an inspection checkpoint rather than a one-shot gate.
- **Error-message wording varied** — Two phrasings appeared: `"name must not exceed 50 characters"` (r0, r1, r5, r6, r8, r9) vs `"name must not be longer than 50 characters"` (r2, r3, r7). r4's report did not pin the exact string. Both forms are functionally equivalent and within-spec.
- **Spec "accepted names" clause phrasing varied** — Examples: "any non-empty length up to 50 characters" (r5, r6, r8), "any non-empty name of 50 characters or fewer" (r4, r7), "names between 1 and 50 characters in length" (r3), "names of any non-empty length up to 50 characters" (r0, r2). All are semantically correct; drift does not constrain spec phrasing, which is appropriate but worth noting.
- **No run made the spec *worse*** (no regressions, no over-deletions, no spec/code mismatches post-reset). This is a divergence from what one might expect with a harder task and indicates the current task difficulty is below the failure threshold for this subject population.

## Prioritized recommendations (consolidated)

No report in the batch contained an explicit tool-improvement recommendation. The following are judge-level observations derived from cross-run patterns, listed by inferred priority:

1. **[Medium]** Consider a "semantic contradiction" or "stale claim" hint in `drift diff`. In every run, subjects had to manually recognize that the existing spec phrase "names of any non-empty length are accepted" had been *falsified* by their new code — drift surfaced the marker/code delta but did not flag that natural-language claims elsewhere in the spec text now contradicted the new behavior. All 10 subjects succeeded, but this is the one place where drift could provide assistive value beyond marker tracking. — *Flagged implicitly by the pattern across all 10 runs (latent)*
2. **[Low]** The retained "No other validation is performed" clause was accurate post-edit only by coincidence (both validations remained the only ones). Consider a linter-style warning when this boilerplate survives a substantive validation-rule addition — to catch the case where a subject adds a *new* validation rule but leaves a "no other validation" assertion unmodified. — *Latent observation from r0–r9*
3. **[Low]** Future eval scenarios should be made harder (multi-spec, cross-file drift, ambiguous stale claims, regression-on-unrelated-marker, or no-op spec edits) to surface failure modes. The current task is saturated: 10/10 perfect, 0/10 tool recommendations. The batch gives high confidence that the *happy path* works, but does not probe the failure surface. — *Latent observation across all runs*

## Next steps

The dominant signal from this batch is that drift's core `todo → diff → reset` loop is well-designed and reliably guides subjects to a correct outcome on a single-spec, single-marker, single-file scenario. With 10/10 runs passing all four criteria and zero tool-improvement recommendations raised by either subjects or judges, the tool has saturated this difficulty level.

Recommended next moves for the tool authors:

1. **Do not invest further in polishing the happy path** — it works. The next learning value comes from harder tasks, not from re-confirming this one.
2. **Design the next eval batch around failure modes**: multi-marker specs where one marker drifts and another should not; cross-file drift where the spec and code live in different packages; partial resets; subjects who try to reset before editing the spec (to see if the tool pushes back); subjects who edit the marker boundary rather than the body; and scenarios where the spec contains a stale natural-language claim that requires *judgment* to detect (probe recommendation #1 above).
3. **Prototype the stale-claim detector** (recommendation #1) as the most promising tool-side improvement. Even if 10/10 subjects handled it manually here, the pattern recurs in every run and the current tool provides no scaffolding for it — a `drift diff` annotation like "warning: spec line N may contradict the implemented behavior" would meaningfully raise the floor.
4. **Re-run this exact task at a future date as a regression smoke-test** to confirm that future tool changes do not break the happy path. It is cheap (subjects complete in seconds) and gives a clean signal.
